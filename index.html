基石

发表于 2020-12-19 2.1k 2 分钟

请保证每四天回来复习一次,

并不是简单的浏览复习, 而是带着思考把每个字都读一遍,

如果有新的理解, 可以及时修改.

池化思想:

线程池, 对象池, 数据库连接池, http 连接池
优点: 重复使用, 用完即还,互相之间不影响,整体监控
缺点: 管理池中对象(最大数量, 核心数量,存活时间),管理请求(请求队列,拒绝策略)

缓存思想:

刚被访问过的数据, 大概率还会被再次访问,如果结果是经过复杂计算(数据库读取,联表查询,第三方接口获取)后得到的值, 短时间内不会发生改变,那么可以将这个值缓存起来,加快访问. 相同输入,相同输出. 依据是 二八定律

中间件思想(策略模式):

对接一个第三方系统时, 不要直接对接,将对接第三方系统的功能抽象为一个接口, 对接时实现这个接口;当这个第三方接口不能用,或者想要增加实现方式时, 继续实现此接口即可.

模板思想:

继承, 抽象类, 接口
准备一个模板, 里面定义了一些需要实现的功能, 有些甚至给出了默认实现, 根据模板生成实例, 在不同的场景下使用不同的实例, 替换增加方便

单一职责思想:

一个组件最好只做一个事情, 并不是做的越多越好. 要足够的高内聚, 这样被复用的几率才会更高, 出错了影响面也会更小, 修改时也更好修改.

事务

ACID 原子性 一致性 隔离性 持久性.
事务是一个抽象层，允许应用程序假装某些并发问题和某些类型的硬件和软件故障不存在。各式各样的错误被简化为一种简单情况：事务中止（transaction abort），而应用需要的仅仅是重试。
当需要保证一个功能必须正常的结束时, 可以采用事务的概念进行封装. begin, commit/rollback.

多版本并发控制 MVCC

快照隔离

分布式事务

调用 A B C 三个系统的接口来系统完成一个操作, 怎么保证一定成功呢?
调用 A 成功, B 失败, 如何回滚?
调用前先 pre check and lock, 保证业务上没有执行问题. 再次执行时, 如果还是失败,只能说明系统出了问题, 需要人工介入. [pre check and lock ,再执行]思想类似于二阶段提交.
目前大多数系统不支持回滚, 支持了幂等的能力, 依然可以让你安全的重试以完成任务. 但是错误数据不会被系统清楚,需要手动处理

MyISAM和InnoDB区别

MyISAM是MySQL的默认数据库引擎（5.5版之前）。虽然性能极佳，而且提供了大量的特性，包括全文索引、压缩、空间函数等，但MyISAM不支持事务和行级锁，而且最大的缺陷就是崩溃后无法安全恢复。不过，5.5版本之后，MySQL引入了InnoDB（事务性数据库引擎），MySQL 5.5版本后默认的存储引擎为InnoDB。
大多数时候我们使用的都是 InnoDB 存储引擎，但是在某些情况下使用 MyISAM 也是合适的比如读密集的情况下。（如果你不介意 MyISAM 崩溃恢复问题的话）。

Java 多线程总结

发表于 2018-02-01 分类于 技术 2k 2 分钟

多线程

线程的状态转换

  • 新建 -> 就绪 -> 运行 -> 阻塞 -> 等待 -> 死亡

如何新启动一个线程

  • implement runnable, implement callable, extend Thread

线程池 ThreadPoolExecutor

  • 使用线程池来统一管理线程,增加线程使用率,减少线程创建时间
  • 构造线程池参数: 核心线程数量,最大线程数量,多余线程存活时间,任务队列,拒绝策略(拒绝抛错,父线程执行,丢弃,丢弃最开始的任务)
  • 提交一个任务到线程池:提交任务 -> 核心线程池是否已满 -> 队列是否已满 -> 最大线程池是否已满 -> 拒绝策略
  • 线程池如何复用线程? 核心线程创建后, 一直while循环获取队列中的任务,超过存活时间后销毁
  • 线程池大小确定: cpu密集型任务, (N+1) IO密集型任务 2N N=CPU核心数

常见线程池 不建议使用, 建议使用原生构造函数自己构造

  • FixedThreadPool 可重用固定线程池(不推荐,无界队列)
  • CachedThreadPool根据需要创建新线程的线程池(无限核心线程)
  • SingleThreadExecutor 只有一个线程的线程池
  • ScheduledThreadPoolExecutor(定期执行任务)

线程中断

  • 中断机制是一种协作机制，也就是说通过中断并不能直接终止另一个线程，而需要被中断的线程自己处理中断。
  • 如何中断? thread.interrupt()
  • 检查中断标志,在线程执行任务时,应该检查中断标志,以应对中断请求
  • 捕获中断异常,在线程被阻塞时,应该捕获中断异常,以应对阻塞时的中断

sleep join wait/notify

  • sleep 不释放锁.一般用于测试时间过去多久之后的状态
  • join a.join 会等待a线程执行结束后再执行主线程
  • wait notify notifyAll 对于某一个object,拿到锁的可以wait,然后唤醒另一个,或者唤醒全部,另一个拿到锁可以执行
  • 生产者消费者实现, 同上可以实现 当队列了之后,prod 调用wait, 当队列空了之后,custom 调用wait notify 唤醒prod

volatile

  • 线程内存会把变量从主内存复制一份到工作内存中,当多线程并发修改同一个变量时, 线程间会丢失修改,造成内存间的不可见
  • 保证内存间可见性: 每次对volatile修饰的变量进行读写时,每个线程都会直接操作主内存中的变量,从而保证了内存间的可见性
  • 某些情况下禁止指令重排打到内存建可见性
  • 仅保证内存间可见性是不能保证线程安全的,例如:都读取到同一个值后,同时更新并写入主内存,就会有部分线程丢失更新

synchronized

  • 作用于: 代码块(同步代码块),方法(静态/非静态: 类的当前实例)
  • 原理: 加锁时,对象头上count+1, 释放锁时 对象头上count-1
  • 互斥锁(只能有一个对象抢占到锁) 可重入锁(外层抢到锁后,指定内层方法不需要再抢锁).
  • 锁优化 自旋锁 -> 偏向锁 -> 轻量级锁 -> 重量级锁 (锁粗化(扩大锁的范围),锁消除(减少不必要的锁))
  • 自旋锁: 进入等待队列之前,自旋很短的时间获取锁
  • 偏向锁: 解决无竞争下的锁性能问题,长时间没有其他线程来抢锁, 这个锁就会一直给当前对象
  • 轻量级锁: 在无竞争条件下,第一次获取锁时采用cas 自旋一下,会很快的获取锁 用于 偏向锁(虽然被抢占,但是没有在使用) -> 轻量级锁
  • 重量级锁: 自旋失败后, 升级重量级锁, 线程阻塞, 进入等地队列, 等待锁的释放

ReentrantLock

  • tryLock unLock tryLock(timeout)
  • 公平锁, 非公平锁 new ReentrantLock(true)

JUC/AQS AbstractQueuedSynchronize

  • 代码层面实现的锁,acquire,acquireQueued,release
  • 队列同步器: 构建锁或者其他同步组件的基础框架（如ReentrantLock、ReentrantReadWriteLock、Semaphore等)
  • 拿到锁的执行代码, 没拿到锁的进入队列排队, 队首就是获得锁的对象,队首执行成功后,唤醒下一个对象获取锁
  • 独占锁(ReentrantLock)
  • 共享锁(CountDownLatch,ReadWriteLock)
  • CountDownLatch: 用来控制一个或多个线程执行完成后进行下一步
  • CyclicBarrier: 用来控制多个线程互相等待, 循环屏障
  • Semaphore: 信号量,acquire() +1, release() -1,限制并发请求数量
  • FutureTask: 可以拿到线程的返回值,通过获取当前线程的状态做操作
  • BlockingQueue: 阻塞队列, 到达队列最大值时, put方法会被阻塞
  • ForkJoin: 双端队列, 工作窃取, 大任务拆分成并行的小任务
  • ThreadLocal: 线程变量副本, 每个线程都有自己的变量副本, 可用于线程执行过程中的参数传递

hadoop 是什么？

hadoop 包括三个部分:

  1. HDFS
  2. Yarn
  3. MapReduce

HDFS

HDFS 是一个分布式，多副本，可以横向扩展的存储系统；可以用 HDFS 开放出来的接口，把分布式文件系统当做本地磁盘一样读取文件，存放文件。
HDFS 对应两个进程 namenode 和 datanode；namenode 存储文件信息，datanode 存储文件本身

Yarn

Yarn 是一个资源调度管框架，它定义了一个任务需要多少资源去运行，需要各种机器相互配合才能完成的大数据任务
yarn 是分布式资源调度框架，如果想要调度你的任务，需要自己实现 yarn 的接口；
yarn 对应两个进程：resourceManager nodeManager。 resourceManager管理着每个node的内存， cpu状态，nodemanager负责上报这些信息

一文搞懂缓存

发表于 2017-10-10 分类于 技术 6k 5 分钟

为什么需要缓存？

  1. 高性能
  2. 高并发

当服务收到大量相同请求，每个请求后端处理起来代价比较大；例如：耗费 cpu 资源、频繁去数据库查询，join 等；
这时候把后端处理到的结果缓存起来，再有相同请求直接缓存中取数据，性能得到提升；并且缓存能承受的 qps 也远大于数据库。

但是问题也随之而来！

  • 当真正的结果改变了，缓存中的值还存在，请求返回的是缓存中的旧值，这样是否业务正确呢？
  • 当你想同步结果到缓存中，怎么保证缓存数据库的双一致性呢？
  • 当缓存挂掉，重启能否恢复缓存数据？如果没有恢复，大量请求到后端数据库，是否需要服务端做些改变？
  • 当你使用的是单机缓存，内存有限，如果设置合适的淘汰策略保证缓存较高的命中率呢？
  • 当你使用的是集群缓存，保证高可用的同时，如何保证各节点间的数据一致性呢？如何在增删机器时，如何避免热点问题呢？
  • 当缓存中间件第一次启动，所有请求都需要后端计算得到，响应速度过慢怎么解决呢？

所以在使用缓存时，如果不是简单缓存（单机且设置固定过期时间），需要把这些问题都考虑进去。
